### A.2a Helper Gap: Sales on Federated Brands

The `get_user_brand_ids(org_id)` function only resolves brands within a single org. When a **sales** user in a rep org needs to create an order for a federated brand (owned by a connected brand org), `get_user_brand_ids` returns nothing for that brand because the user is not a member of the brand org. The current workaround is the separate `get_connected_org_ids()` federation policy on `brands` SELECT, plus the expanded INSERT policy on `brand_expenses` that OR's in `get_connected_org_ids()`. The **orders** INSERT policy still requires `brand_id IN get_user_brand_ids(organization_id)`, meaning a sales user cannot directly insert an order for a federated brand through RLS alone -- the application must set `brand_id` to a brand the user's own org owns (and auto-federation links it to the brand org). This is by design: orders belong to the rep org, not the brand org.

### Accounts federation asymmetry

Accounts (and their satellites: `account_locations`, `account_tags`, `account_tag_assignments`) use **asymmetric** federation, unlike brands/products where the implicit `get_connected_org_ids()` policy is symmetric:

- **Rep→Brand (implicit):** members of a rep org see accounts owned by any brand org they have an active `org_connections` row with. Rep-side-only subquery on `org_connections` — brand org members do NOT benefit from this policy. Implemented by `20260418000012_rep_sees_connected_brand_accounts.sql`.
- **Brand→Rep (explicit):** brand org members see a rep's account ONLY via a `federated_account_links` row (target_org_id = brand org). These rows are created by the `auto_federate_order()` trigger when a rep places an order against one of the brand's brands. Implemented in `20260411000002_federation_infrastructure.sql`.

Rationale: a rep has no privacy interest in hiding which brands they rep for (those connections are mutually known), but a rep has a strong interest in NOT exposing their full account book to every connected brand. A brand should only see the rep's accounts that have actually placed an order with it.

## A.6 Federation Direction Cheat-Sheet

> **ADR:** [`docs/adr/0001-federation-model.md`](../adr/0001-federation-model.md) — records the accepted decision and rationale for one federation model per direction.

Federation is **asymmetric**. The two directions use different mechanisms.

### MBISR (rep org) sees BOA (brand org) data

**Direction:** Rep -> Brand

**Mechanism:** Implicit via `get_connected_org_ids()`

**What reps see from connected brands:**

| Data               | How                                                                                                                                   |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| Brands             | `brands` SELECT policy: `organization_id IN get_connected_org_ids()`                                                                  |
| Brand assets       | `brand_assets` SELECT policy: `organization_id IN get_connected_org_ids()`                                                            |
| Products           | `products` SELECT policy: `organization_id IN org_connections.brand_org_id`                                                           |
| Product variants   | `product_variants` SELECT policy: via products join + `org_connections`                                                               |
| Product images     | `product_images` SELECT policy: via products join + `org_connections`                                                                 |
| Accounts (brand's) | `accounts` SELECT policy: `organization_id IN (rep_side org_connections.brand_org_id, active)` — unidirectional; see `20260418000012` |
| Account locations  | `account_locations` SELECT policy: same rep-side-only subquery                                                                        |
| Account tags       | `account_tags` SELECT policy: same rep-side-only subquery                                                                             |
| Brand expenses     | `brand_expenses` SELECT: `brand_id` in connected org's brands. INSERT: allowed for federated brands                                   |

**What reps CANNOT do to brand data:**

- Cannot INSERT/UPDATE/DELETE brands, brand_assets, products, product_variants, product_images in the brand org
- Cannot modify accounts/account_locations in the brand org
- Cannot see brand-org-internal tables (seasons, territories, shows, team members, integrations)

### BOA (brand org) sees MBISR (rep org) data

**Direction:** Brand -> Rep

**Mechanism:** Explicit via `federated_order_links` and `federated_account_links`

**What brands see from connected reps:**

| Data                   | How                                                                        |
| ---------------------- | -------------------------------------------------------------------------- |
| Orders                 | `orders` SELECT policy: `id IN federated_order_links` (target_org, active) |
| Order lines            | `order_lines` SELECT policy: `order_id IN federated_order_links`           |
| Order comments         | `order_comments` SELECT + INSERT: via `federated_order_links`              |
| Order audits           | `order_audits` SELECT: via `federated_order_links`                         |
| Accounts (rep's)       | `accounts` SELECT: `id IN federated_account_links` (target_org)            |
| Rep org accounts (all) | `accounts` SELECT: `organization_id IN get_connected_org_ids()`            |

**What brands CAN do to rep data:**

- UPDATE order status on federated orders (special policy: `Brand updates federated order status`)
- INSERT comments on federated orders

**What brands CANNOT do:**

- Cannot create orders in the rep org
- Cannot modify rep accounts
- Cannot see rep-org-internal tables (territories, shows, team members, integrations, expenses)

### Auto-federation trigger

When a rep creates an order for a connected brand, the `auto_federate_order()` trigger automatically:

1. Sets `connection_id` on the order
2. Creates a `federated_order_links` row (source=rep org, target=brand org)
3. Creates a `federated_account_links` row (if not already linked)

This means brand visibility of orders/accounts is **opt-in by the act of creating an order**, not blanket access to all rep orders.
