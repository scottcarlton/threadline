# /Threadline — Email Templates

Branded transactional templates for the 9 Brevo emails Threadline sends today, plus 5 Supabase Auth emails. Built against `docs/brand/guidelines.md`.

## Folder layout

```
emails/
  README.md                          ← this file
  _shell.html                        ← reference shell (header / body / footer) — do not paste, see notes
  template-ids.json                  ← slug → Brevo numeric template ID mapping (generated by upload script)
  brevo/
    invite-org-member.html
    invite-buyer-team.html
    invite-connection-member.html
    invite-org-connect.html

  auth/

    reauthentication.html
  text/                              ← plain-text fallbacks (one per template)
```

## Brand decisions baked in

- Wordmark `/Threadline` — never split, never recolored
- Colors — Background `#FCFCFC`, Foreground `#292B30`, Primary `#181820`, Muted text `#717179`
- Typography — IBM Plex Sans (body), IBM Plex Mono (labels) loaded from Google Fonts with system fallbacks
- Sharp corners everywhere (`border-radius: 0`)
- Mono labels in brackets (`[ ORDER SUBMITTED ]`) as the canonical section marker
- Two-color, no decorative color, no gradients, no shadows, no emojis
- Voice: direct, lowercase headlines where natural, no superlatives, no exclamation marks, industry language used naturally (line sheets, sell-through, account)

## How to install in Brevo

The 9 templates here are uploaded as Brevo SMTP Templates by running:

```bash
BREVO_API_KEY=<key> bun run scripts/upload-brevo-templates.ts
```

The script is upsert-safe — it checks `emails/template-ids.json` for existing IDs (updates by ID) or falls back to searching by name (creates if new). Requires `BREVO_API_KEY` (full access permission) in the environment.

After uploading, the script writes `emails/template-ids.json` with the slug → numeric ID mapping. Callers reference templates by ID:

```ts
await sendEmail({
  to,
  subject: 'Order confirmed',
  html: '',
  templateId: 15, // from template-ids.json
  params: {
    ORDER_NUMBER: order.number,
    ACCOUNT_NAME: account.business_name,
    BRAND_NAME: brand.name,
    TOTAL: formatCurrency(order.total_amount),
    ORDER_URL: `${env.PUBLIC_APP_URL}/orders/${order.id}`
  },
  template: 'order-confirmed'
});
```

### Brevo syntax

Brevo uses Twig-style tags:

- Variables: `{{params.VARIABLE_NAME}}` (UPPER_SNAKE_CASE)
- Conditionals: `{% if params.VAR %}...{% else %}...{% endif %}`
- Loops: `{% for item in params.ITEMS %}...{% endfor %}`

Variable names are uppercased (e.g. `ORDER_NUMBER`). The `params.` prefix is required.

### Alternative: inline HTML, no Brevo Templates

If you'd rather ship without using Brevo Templates at all, copy each `brevo/<slug>.html` body into `src/lib/server/email-templates.ts` and swap `{{params.VAR}}` for `${escapeHtml(var)}` template literals. The `sendEmail()` callsites stay unchanged — just omit `templateId`.

## Variables, by template

### invite-org-member · invite-buyer-team · invite-connection-member
- `INVITER_NAME`
- `ORGANIZATION_NAME` (or business / account name for the buyer-team variant)
- `ROLE` — admin / member / sales / guest / buyer (omit if not applicable)
- `ACCEPT_URL`
- `EXPIRES_IN_DAYS` (default `7`)

### invite-org-connect
- `FROM_ORG_NAME`
- `FROM_USER_NAME` (optional — empty string hides the line)
- `PERSONAL_MESSAGE` (optional — empty string hides the quote block)
- `INVITE_URL`

### order-submitted · order-created · order-confirmed · order-shipped · order-delivered
- `ORDER_NUMBER`
- `ACCOUNT_NAME`
- `BRAND_NAME`
- `TOTAL` — pre-formatted currency string (e.g. `$4,200`)
- `LINE_COUNT` (optional)
- `ORDER_URL`
- `TRACKING_NUMBER` (shipped only, optional)
- `TRACKING_URL` (shipped only, optional)

### auth/* (Supabase)
Use Supabase's variable names — already in place:
- `{{ .ConfirmationURL }}`
- `{{ .Token }}`
- `{{ .TokenHash }}`
- `{{ .SiteURL }}`
- `{{ .Email }}`
- `{{ .NewEmail }}` (change-email only)

Paste each into Supabase Dashboard → Authentication → Email Templates.

## Plain-text fallbacks

Every template has a `text/<slug>.txt` partner. These are used when sending inline HTML via `sendEmail({ html, text })`. When using Brevo Templates (`templateId`), the plain-text version is managed within Brevo's template editor.

## Subject lines & preheaders

Each HTML file has its recommended subject line and preheader at the top inside `<!-- subject: ... -->` and `<!-- preheader: ... -->` HTML comments. The upload script sets the subject from its `TEMPLATES` array — update subjects there, not in the HTML comments (the comments are for human reference only).

## Browser-rendered preview

Open any `.html` file directly in a browser to see exactly how it renders. The CSS is inlined and the Google Fonts `<link>` is included. Twig variables show as literal text — that's expected.

## Dark-mode behavior

Threadline brand emails are intentionally fixed in light mode. The templates set explicit colors and use `meta name="color-scheme" content="only light"` so Apple Mail and Outlook don't auto-invert them. This protects the wordmark contrast and the foreground/background relationship.

## Updating the shell

`_shell.html` is the canonical layout (header, footer, button, mono label). When you edit shell-level styling, regenerate every template — or, if you move to React Email later, lift the shell into `<EmailLayout>` and re-render.
